OSCAL Assessment Plan JSON Schema Reference

Schema download oscal_assessment-plan_schema.json

Schema version 1.0.0-milestone3

Short name oscal-ap

The OSCAL assessment plan format is used to describe the information typically provided by an assessor during the preparation for an assessment.

The root of the OSCAL assessment plan format is assessment-plan.

activity-uuid

Name Activity ID

Description Links the task to a defined activity.

A string property (type uuid).

This object appears as a property on: task.

addr-line

Name Address line

Description A single line of an address.

A string property.

This object appears as a property on: address.

address

Name Address

Description A postal address.

This object appears as a property on: location, party.

The address object has the following members (properties):

  • type property string (optional) Indicates the type of address.

    The value may be locally defined, or one of the following:

    • home: A home address.
    • work: A work address.

  • An array labeled postal-address, containing addr-line objects, labeled postal-address

    [0 to ∞] A single line of an address.

  • city object [0 or 1]

    City, town or geographical region for mailing address

  • state object [0 or 1]

    State, province or analogous geographical region for mailing address

  • postal-code object [0 or 1]

    Postal or ZIP code for mailing address

  • country object [0 or 1]

    Country for mailing address

algorithm

Name Hash algorithm

Description Method by which a hash is derived

A string property.

The value may be locally defined, or one of the following:

  • SHA-224
  • SHA-256
  • SHA-384
  • SHA-512
  • SHA3-224
  • SHA3-256
  • SHA3-384
  • SHA3-512

Appears as a property on: hash.

Remarks

Valid values MUST be a value defined in the W3C XML Security Algorithm Cross-Reference Digest Methods (W3C, April 2013).

all

Name All

Description A key word to indicate all

A string property.

This object appears as a property on: controls, control-objectives, include-subject, exclude-subject.

annotation

Name Annotation

Description A name/value pair with optional explanatory remarks.

This object appears as a property on: location, party, role, responsible-party, responsible-role, user, component, inventory-item, implemented-component, objectives, controls, control-objectives, objective, method, include-subject, exclude-subject, assets, test-method, task, include-activity, exclude-activity.

The annotation object has the following members (properties):

  • name property NCName (required)
  • uuid property uuid (required)
  • ns property string (optional)
  • value property string (optional) Indicates the value of the characteristic.

  • remarks object [0 or 1]

    Additional commentary on the parent item.

Remarks

Annotations permit the deployment and management of arbitrary controlled values for any purpose useful to an application or implementation scenario.

The value (flag) of an annotation is a simple scalar value or list of values. The body of the annotation offers remarks in explanation of or reference to this value.

assessment-activities

Name Assessment Activities

Description Identifies the assessment activities and schedule. In the assessment plan, these are planned activities. In the assessment results, these are the actual activities performed.

This object appears as a property on: assessment-plan.

The assessment-activities object has the following members (properties):

  • An array labeled test-methods, containing test-method objects, labeled test-methods

    [0 to ∞] Identifies an individual test method.

  • schedule object [0 or 1]

    Identifies the schedule for the assessment activities.

  • An array labeled include-activities, containing include-activity objects, labeled include-activities

    [0 to ∞] Identifies an assessment activity. In the assessment plan, this is an intended/in-scope activity. In the assessment results, this identifies an activity that was actually performed.

  • An array labeled exclude-activities, containing exclude-activity objects, labeled exclude-activities

    [0 to ∞] Identifies an activity explicitly excluded from the assessment. In the assessment plan, this clarifies activities that are out-of-scope or prohibited. In the assessment results, this could be used to explicitly identify an activity that was planned, but not performed.

  • remarks object [0 or 1]

    Additional commentary on the parent item.

assessment-plan

Name Security Assessment Plan (SAP)

Description An assessment plan, such as those provided by a FedRAMP assessor.

assessment-plan is the root (containing) object of this schema.

The assessment-plan object has the following members (properties):

  • uuid property uuid (required)

  • metadata object [1]

    Provides information about the publication and availability of the containing document.

  • import-ssp object [1]

    Used by the SAP to import information about the system being assessed.

  • objectives object [1]

    Identifies the controls and control being assessed and their control objectives. In the assessment plans, these are the planned controls and objectives. In the assessment results, these are the actual controls and objectives, and reflects any changes from the plan.

  • assessment-subjects object [0 or 1]

    Identifies system elements being assessed, such as components, inventory items, and locations. In the assessment plan, this identifies the planned assessment subject. In the assessment results this is the actual assessment subject, and reflects any changes from the plan.

  • assets object [0 or 1]

    Identifies the assets used to perform this assessment, such as the assessment team, scanning tools, and assumptions.

  • assessment-activities object [0 or 1]

    Identifies the assessment activities and schedule. In the assessment plan, these are planned activities. In the assessment results, these are the actual activities performed.

  • back-matter object [0 or 1]

    A collection of citations and resource references.

assessment-subjects

Name Subject of Assessment

Description Identifies system elements being assessed, such as components, inventory items, and locations. In the assessment plan, this identifies the planned assessment subject. In the assessment results this is the actual assessment subject, and reflects any changes from the plan.

This object appears as a property on: assessment-plan.

The assessment-subjects object has the following members (properties):

  • An array labeled includes, containing include-subject objects, labeled includes

    [1 to ∞] Identifies exactly what will be the focus of this assessment. Anything not explicitly defined is out-of-scope.

  • An array labeled excludes, containing exclude-subject objects, labeled excludes

    [0 to ∞] Identifies what is explicitly excluded from this assessment. Used to remove a subset of items from groups of explicitly included items. Also used to explicitly clarify off-limit items, such as hosts to avoid scanning.

  • local-definitions object [0 or 1]

    Allows control objectives, users, components, and inventory-items to be defined within the assessment plan or assessment results for circumstances where they are not appropriately defined in the SSP. NOTE: Use the assessment plan or assessment results metadata to define additional locations if needed.

  • remarks object [0 or 1]

    Additional commentary on the parent item.

assets

Name Assessment Assets

Description Identifies the assets used to perform this assessment, such as the assessment team, scanning tools, and assumptions.

This object appears as a property on: assessment-plan.

The assets object has the following members (properties):

  • tools object [0 or 1]

    The technology tools used by the assessor to perform the assessment, such as vulnerability scanners. In the assessment plan these are the intended tools. In the assessment results, these are the actual tools used, including any differences from the assessment plan.

  • origination object [0 or 1]

    Identifies the origination of network-based assessment activities, such as the IP address of the tool performing assessment scans.

  • An array labeled properties, containing prop objects, labeled properties

    [0 to ∞] Provided as means of extending the OSCAL syntax.

  • An array labeled annotations, containing annotation objects, labeled annotations

    [0 to ∞] Provided as means of extending the OSCAL syntax.

  • An array labeled parts, containing part objects, labeled parts

    [0 to ∞] Provided to allow content such as assumptions and disclosures.

  • remarks object [0 or 1]

    Additional commentary on the parent item.

back-matter

Name Back matter

Description A collection of citations and resource references.

This object appears as a property on: assessment-plan.

The back-matter object has the following members (properties):

  • An array labeled resources, containing resource objects, labeled resources

    [0 to ∞] A resource associated with the present document, which may be a pointer to other data or a citation.

Remarks

Provides a collection of identified citation and resource objects that can be referenced by a link with a rel value of "reference" and an href value that is a fragment "#" followed by a reference to a citation identifier. A citation can reference a resource by specifying a local target value using the same fragment-identifier approach.

The following is a contrived example to show the use of link, citation, and resource.

Example for 'back-matter' not JSONized -- 
            #resource1citationMy citation
            https://example.org/some-resource

Name Bibliographic Definition

This object appears as a property on: citation.

The biblio object has the following members (properties):

    citation

    Name Citation

    Description A citation consisting of end note text and optional structured bibliographic data.

    This object appears as a property on: resource.

    The citation object has the following members (properties):

    • text object [1]

      A line of textual content whose semantic is determined by the context of use.

    • An array labeled properties, containing prop objects, labeled properties

      [0 to ∞] A value with a name, attributed to the containing control, part, or group.

    • biblio object [0 or 1]

      A container in which a set of bibliographic information can included. The model of this information is undefined by OSCAL.

    Remarks

    A text is used to define the endnote text, without any explicit bibliographic structure. If structured bibliographic data is needed, then the biblio> can be used for this purpose.

    A biblio can be used to capture a structured bibliograpical citation in an appropriate format.

    Name Compare To

    Description Typically used in when copying content from the assessment plan to the assessment results. The uuid should be changed in the assessment results file, and the compare-to field should be set to the original assessment plan uuid value.     component

    Name Component

    Description A defined component that can be part of an implemented system.

    This object appears as a property on: tools, local-definitions.

    The component object has the following members (properties):

    • Implicitly, a property uuid (type uuid): this value will appear as the key (label) of the parent component JSON object

      The unique identifier for the component.
    • component-type property string (required)

      The value may be locally defined, or one of the following:

      • software: Any software, operating system, or firmware
      • hardware
      • service: A service that may provide APIs.
      • interconnection: A connection to something outside this system.
      • policy
      • process
      • procedure
      • plan
      • guidance: Any guideline or recommendation
      • standard
      • validation: An external assessment performed on some other component, that has been validated by a third-party.

    • title object [1]

      A human readable name for the system component.

    • description object [1]

      A description of the component, including information about its function.

    • purpose object [0 or 1]

      A summary of the technological or business purpose of the component.

    • An array labeled properties, containing prop objects, labeled properties

      [0 to ∞] A value with a name, attributed to the containing control, part, or group.Some specific property names can be assigned to the component at large, or can be defined when a component is associated with an inventory-item within the implemented-component construct.

    • An array labeled annotations, containing annotation objects, labeled annotations

      [0 to ∞] A name/value pair with optional explanatory remarks.Some specific annotation names can be assigned to the component at large, or can be defined when a component is associated with an inventory-item within the implemented-component construct.

    • An array labeled links, containing link objects, labeled links

      [0 to ∞] A reference to a local or remote resource

    • status object [1]

      Describes the operational status of the system.

    • Labeled responsible-role objects [0 to ∞]

      Defines a role that has responsibility for the component.A role defined in this way can be associated with a party or parties here, or where the component is referenced (e.g., inventory-item, implemented-requirement/by-component).The roles "maintainer" or "provider" can be used to define the supplier. In such cases party-id(s) can be used to associate the providing organization. For products, services, and APIs, this is the vendor and/or retailer information. For policies, processes, plans, standards, and guidance, this is the authoring organization.

    • An array labeled protocols, containing protocol objects, labeled protocols

      [0 to ∞] Used for service components to define the protocols supported by the service.

    • remarks object [0 or 1]

      Additional commentary on the parent item.

    Remarks

    Components may be products, services, application programming interface (APIs), policies, processes, plans, guidance, standards, or other tangible items that enable security and/or privacy.

    The type indicates which of these component types is represented.

    When defining a service compoenent where are relationship to other components is known, one or more link entries with rel values of provided-by and used-by can be used to link to the specific component identifer(s) that provide and use the service respectively.

    Name Document Identifier

    Description A document identifier qualified by an identifier type.

    An object with property identifier

    This object appears as a property on: metadata, resource.

    The doc-id object must additionally have a (string) property:

    • type property string (required) Qualifies the kind of document identifier.

      The value may be locally defined, or one of the following:

      • doi: A Digital Object Identifier (DOI); use is preferred, since this allows for retrieval of a full bibliographic record.

    Remarks

    A document identifier or name, qualified by a type that indicates the naming system of the identifier. A document identifier may provide a hint to the persistent location of the same or another version of the document.

    Specific usage rules, nomenclatures, and recognized file types (e.g., DOIs, CMS locators, etc.) may be defined at application level. The value of this field may also be a hash or checksum.

    Description A partition or component of a control or part

    • name property NCName (required)

    Remarks

    A part provides for logical partitioning of prose, and can be thought of as a grouping structure (e.g., section). A part can have child parts allowing for arbitrary nesting of prose content (e.g., statement hierarchy). A part can contain prop objects that allow for enriching prose text with structured name/value information.

    A part can be assigned an optional uuid, which allows for internal and external references to the textual concept contained within a part. A id provides a means for an OSCAL profile, or a higher layer OSCAL model to reference a specific part within a catalog. For example, an id can be used to reference or to make modifications to a control statement in a profile.

    Use of part and prop provides for a wide degree of extensibility within the OSCAL catalog model. The optional ns provides a means to qualify a part's name, allowing for organization-specific vocabularies to be defined with clear semantics. Any organization that extends OSCAL in this way should consistently assign a ns value that represents the organization, making a given namespace qualified name unique to that organization. This allows the combination of ns and name to always be unique and unambiguous, even when mixed with extensions from other organizations. Each organization is responsible for governance of their own extensions, and is strongly encouraged to publish their extensions as standards to their user community. If no ns is provided, the name is expected to be in the "OSCAL" namespace.

    To ensure a ns is unique to an organization and naming conflicts are avoided, a URI containing a DNS or other globally defined organization name should be used. For example, if FedRAMP and DoD both extend OSCAL, FedRAMP will use the ns "https://fedramp.gov", while DoD will use the ns "https://defense.gov" for any organization specific name.

    Tools that process OSCAL content are not required to interpret unrecognized OSCAL extensions; however, OSCAL-compliant tools should not modify or remove unrecognized extensions, unless there is a compelling reason to do so, such as data sensitivity.
