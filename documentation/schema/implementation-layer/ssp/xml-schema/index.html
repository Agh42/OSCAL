OSCAL System Security Plan (SSP) Model XML Schema Reference

Schema download oscal_ssp_schema.xsd

Schema version 1.0.0-milestone3

Short name oscal-ssp

XML namespace http://csrc.nist.gov/ns/oscal/1.0

The OSCAL Control SSP format can be used to describe the information typically specified in a system security plan, such as those defined in NIST SP 800-18.

The root of the OSCAL System Security Plan (SSP) format is system-security-plan.

addr-line

Name Address line

Description A single line of an address.

An element (string)

This element appears inside: address.

address

Name Address

Description A postal address.

This element appears inside: location and party.

The address element has the following contents (in order):

  • type attribute (string) [optional] Indicates the type of address.

    The value may be locally defined, or one of the following:

    • home: A home address.
    • work: A work address.

  • addr-line elements (string) [0 to ∞] A single line of an address.

  • city element (string) [0 or 1] City, town or geographical region for mailing address

  • state element (string) [0 or 1] State, province or analogous geographical region for mailing address

  • postal-code element (string) [0 or 1] Postal or ZIP code for mailing address

  • country element (string) [0 or 1] Country for mailing address

adjustment-justification

Name Adjustment Justification

Description If the selected security level is different from the base security level, this contains the justification for the change.

An element (markup-multiline)

This element appears inside: confidentiality-impact, integrity-impact, and availability-impact.

algorithm

Name Hash algorithm

Description Method by which a hash is derived

An attribute (string)

The value may be locally defined, or one of the following:

  • SHA-224
  • SHA-256
  • SHA-384
  • SHA-512
  • SHA3-224
  • SHA3-256
  • SHA3-384
  • SHA3-512

This attribute appears on: hash.

Remarks

Valid values MUST be a value defined in the W3C XML Security Algorithm Cross-Reference Digest Methods (W3C, April 2013).

annotation

Name Annotation

Description A name/value pair with optional explanatory remarks.

This element appears inside: location, party, role, responsible-party, responsible-role, system-characteristics, system-information, leveraged-authorization, authorization-boundary, network-architecture, data-flow, system-implementation, user, component, inventory-item, implemented-component, implemented-requirement, statement, by-component, export, provided, responsibility, inherited, and satisfied.

The annotation element has the following contents:

  • name attribute (NCName) [required]

  • uuid attribute (uuid) [required]

  • ns attribute (string) [optional]

  • value attribute (string) [optional] Indicates the value of the characteristic.

  • remarks element (markup-multiline) [0 or 1] Additional commentary on the parent item.

Remarks

Annotations permit the deployment and management of arbitrary controlled values for any purpose useful to an application or implementation scenario.

The value (flag) of an annotation is a simple scalar value or list of values. The body of the annotation offers remarks in explanation of or reference to this value.

availability-impact

Name Availability Impact Level

Description The expected level of impact resulting from the disruption of access to or use of information or the information system.

This element appears inside: information-type.

The availability-impact element has the following contents (in order):

  • prop elements (string) [0 to ∞] A value with a name, attributed to the containing control, part, or group.

  • base element (string) [1] The prescribed base (Confidentiality, Integrity, or Availability) security impact level.

    The value must be one of the following:

    • fips-199-low: A 'low' sensitivity level as defined in FIPS-199.
    • fips-199-moderate: A 'moderate' sensitivity level as defined in FIPS-199.
    • fips-199-high: A 'high' sensitivity level as defined in FIPS-199.

  • selected element (string) [0 or 1] The selected (Confidentiality, Integrity, or Availability) security impact level.

    The value must be one of the following:

    • fips-199-low: A 'low' sensitivity level as defined in FIPS-199.
    • fips-199-moderate: A 'moderate' sensitivity level as defined in FIPS-199.
    • fips-199-high: A 'high' sensitivity level as defined in FIPS-199.

  • adjustment-justification element (markup-multiline) [0 or 1] If the selected security level is different from the base security level, this contains the justification for the change.

back-matter

Name Back matter

Description A collection of citations and resource references.

This element appears inside: system-security-plan.

The back-matter element has the following contents:

  • resource elements [0 to ∞] A resource associated with the present document, which may be a pointer to other data or a citation.

Remarks

Provides a collection of identified citation and resource objects that can be referenced by a link with a rel value of "reference" and an href value that is a fragment "#" followed by a reference to a citation identifier. A citation can reference a resource by specifying a local target value using the same fragment-identifier approach.

The following is a contrived example to show the use of link, citation, and resource.

<profile>
<metadata>
<link rel="citation" href="#resource1">My citation</link></metadata>
<back-matter>
<resource id="resource1">
<rlink href="https://example.org/some-resource"></rlink></resource></back-matter></profile>

Name Citation

Description A citation consisting of end note text and optional structured bibliographic data.

This element appears inside: resource.

The citation element has the following contents (in order):

  • text element (markup-line) [1] A line of textual content whose semantic is determined by the context of use.

component

Name Component

Description A defined component that can be part of an implemented system.

This element appears inside: system-implementation.

The component element has the following contents (in order):

  • uuid attribute (uuid) [required] The unique identifier for the component.

  • component-type attribute (string) [required]

    The value may be locally defined, or one of the following:

    • software: Any software, operating system, or firmware
    • hardware
    • service: A service that may provide APIs.
    • interconnection: A connection to something outside this system.
    • policy
    • process
    • procedure
    • plan
    • guidance: Any guideline or recommendation
    • standard
    • validation: An external assessment performed on some other component, that has been validated by a third-party.

  • title element (markup-line) [1] A human readable name for the system component.

  • description element (markup-multiline) [1] A description of the component, including information about its function.

  • purpose element (markup-line) [0 or 1] A summary of the technological or business purpose of the component.

  • prop elements (string) [0 to ∞] A value with a name, attributed to the containing control, part, or group.

    Some specific property names can be assigned to the component at large, or can be defined when a component is associated with an inventory-item within the implemented-component construct.

  • annotation elements [0 to ∞] A name/value pair with optional explanatory remarks.

    Some specific annotation names can be assigned to the component at large, or can be defined when a component is associated with an inventory-item within the implemented-component construct.

  • link elements (markup-line) [0 to ∞] A reference to a local or remote resource

  • status element [1] Describes the operational status of the system.

  • responsible-role elements [0 to ∞] Defines a role that has responsibility for the component.

    A role defined in this way can be associated with a party or parties here, or where the component is referenced (e.g., inventory-item, implemented-requirement/by-component).

    The roles "maintainer" or "provider" can be used to define the supplier. In such cases party-id(s) can be used to associate the providing organization. For products, services, and APIs, this is the vendor and/or retailer information. For policies, processes, plans, standards, and guidance, this is the authoring organization.

  • protocol elements [0 to ∞] Used for service components to define the protocols supported by the service.

  • remarks element (markup-multiline) [0 or 1] Additional commentary on the parent item.

Remarks

Components may be products, services, application programming interface (APIs), policies, processes, plans, guidance, standards, or other tangible items that enable security and/or privacy.

The type indicates which of these component types is represented.

When defining a service compoenent where are relationship to other components is known, one or more link entries with rel values of provided-by and used-by can be used to link to the specific component identifer(s) that provide and use the service respectively.

diagram

Name Diagram

Description A graphic that provides a visual representation the system, or some aspect of it.

This element appears inside: authorization-boundary, network-architecture, and data-flow.

The diagram element has the following contents (in order):

  • uuid attribute (uuid) [required] The identifier for this diagram.

  • description element (markup-multiline) [0 or 1] A description of the diagram (e.g., alternate text). This can be used to support compliance with requirements from Section 508 of the United States Workforce Rehabilitation Act of 1973.

  • prop elements (string) [0 to ∞] A value with a name, attributed to the containing control, part, or group.

  • link elements (markup-line) [0 to ∞] A reference to a local or remote resource

  • caption element (markup-line) [0 or 1] A brief caption to annotate the diagram.

  • remarks element (markup-multiline) [0 or 1] Commentary about the diagram that enhances it.

Remarks

A diagram must include a link with a rel value of "diagram", who's href references a remote URI or an internal reference within this document containing the diagram.

The internal reference "#diagram1" points to an attached resource defined in the back-matter as a resource. The media-type indicates that the image is a Portable Network Graphics (PNG) image.

<diagram id="boundary-diagram-1">
<description>A boundary diagram.</description>
<link rel="diagram" href="#diagram1" media-type="image/png"></link></diagram>

doc-id

Name Document Identifier

Description A document identifier qualified by an identifier type.

An element (string)

This element appears inside: metadata and resource.

The doc-id element must have the attribute:

  • type attribute (string) [required] Qualifies the kind of document identifier.

    The value may be locally defined, or one of the following:

    • doi: A Digital Object Identifier (DOI); use is preferred, since this allows for retrieval of a full bibliographic record.

Remarks

A document identifier or name, qualified by a type that indicates the naming system of the identifier. A document identifier may provide a hint to the persistent location of the same or another version of the document.

Specific usage rules, nomenclatures, and recognized file types (e.g., DOIs, CMS locators, etc.) may be defined at application level. The value of this field may also be a hash or checksum.

implemented-component

Name Implemented Component

Description The set of componenets that are implemented in a given system inventory item.

This element appears inside: inventory-item.

The implemented-component element has the following contents (in order):

    The value may be locally defined, or one of the following:

    • runs-software: The implemented component is a 'software' component that the inventory item has installed and uses.
    • uses-hardware: The implemented component is a 'hardware' component that the inventory item has installed and uses.

  • prop elements (string) [0 to ∞] A value with a name, attributed to the containing control, part, or group.

  • link elements (markup-line) [0 to ∞] A reference to a local or remote resource

  • responsible-party elements [0 to ∞] A reference to a set of organizations or persons that have responsibility for performing a referenced role relative to the parent context.

    This construct is used to either: 1) associate a party or parties to a role defined on the component using the responsible-role construct, or 2) to define a party or parties that are responsible for a role defined within the context of the containing inventory-item.

  • remarks element (markup-multiline) [0 or 1] Additional commentary on the parent item.

resource

Name Resource

Description A resource associated with the present document, which may be a pointer to other data or a citation.

This element appears inside: back-matter.

The resource element has the following contents (in order):

  • uuid attribute (uuid) [required]

  • title element (markup-line) [0 or 1] A title for display and navigation

  • desc element (string) [0 or 1] A short textual description

  • prop elements (string) [0 to ∞] A value with a name, attributed to the containing control, part, or group.

  • doc-id elements (string) [0 to ∞] A document identifier qualified by an identifier type.

  • citation element [0 or 1] A citation consisting of end note text and optional structured bibliographic data.

  • rlink elements [0 to ∞] A pointer to an external copy of a document with optional hash for verification

  • base64 elements (base64Binary) [0 to ∞]

  • remarks element (markup-multiline) [0 or 1] Additional commentary on the parent item.

  • Any element (in a foreign namespace)

Remarks

Use of resource allows for information that pertains to, authenticates, or supplements the document it is included in.

A resource can be used in two ways. 1) it may point to an specific retrievable network resource using a rlink, and 2) be included as an attachment using a base64. A resource may contain multiple rlink and base64 entries that represent alternative download locations (rlink) and attachments (base64) for the same resource. Both rlink and base64 allow for a media-type to be specified, which is used to distiguish between different representations of the same resource (e.g., Microsoft Word, PDF). When multiple rlink and base64 entries are included for a given resource, all entries must contain equivalent information. This is extremely important when the entries represent OSCAL content that is represented in alternate formats (i.e., XML, JSON, YAML). A processor must be able to choose a preferred format to process (based on media-type) with the assumption that the same data can be processed from any of the available formats included by entries.

When a resource is used as a citation, then the title and citation properties must be included.

Description A target-level of availability for the system, based on the sensitivity of information within the system.

An element (string)

The value must be one of the following:

  • fips-199-low: A 'low' sensitivity level as defined in FIPS-199.
  • fips-199-moderate: A 'moderate' sensitivity level as defined in FIPS-199.
  • fips-199-high: A 'high' sensitivity level as defined in FIPS-199.

This element appears inside: security-impact-level.

security-sensitivity-level

Name Security Sensitivity Level

Description The overall information system sensitivity categorization, such as defined by FIPS-199.

An element (string)

The value must be one of the following:

  • low: A 'low' sensitivity level, such as defined by FIPS-199.
  • moderate: A 'moderate' sensitivity level, such as defined by FIPS-199.
  • high: A 'high' sensitivity level, such as defined by FIPS-199.

This element appears inside: system-characteristics.

Remarks

Often, organizations require the security sensitivity level to correspond with the highest confidentiality, integrity, or availability level identified by security-impact-level.

Description Identifies which statements within a control are addressed.

  • annotation elements [0 to ∞] A name/value pair with optional explanatory remarks.

  • link elements (markup-line) [0 to ∞] A reference to a local or remote resource

  • responsible-role elements [0 to ∞] A reference to one or more roles with responsibility for performing a function relative to the control.

status

Name Status

Description Describes the operational status of the system.

This element appears inside: system-characteristics and component.

The status element has the following contents:

  • state attribute (string) [required] The current operating status.

    The value must be one of the following:

    • operational: The system is currently operating in production.
    • under-development: The system is being designed, developed, or implemented
    • under-major-modification: The system is undergoing a major change, development, or transition.
    • disposition: The system is no longer operational.
    • other: Some other state.

  • remarks element (markup-multiline) [0 or 1] Additional commentary on the parent item.

Remarks

If 'other' is selected, a remark must be included to describe the current state.

system-characteristics

Name System Characteristics

Description Contains the characteristics of the system, such as its name, purpose, and security impact level.

This element appears inside: system-security-plan.

The system-characteristics element has the following contents (in order):

  • system-id elements (string) [1 to ∞] A unique identifier for the system described by this system security plan.

  • system-name element (string) [1] The full name of the system.

  • system-name-short element (string) [0 or 1] A short name for the system, such as an acronym, that is suitable for display in a data table or summary list.

  • description element (markup-multiline) [1] A free-text description of the system.

  • prop elements (string) [0 to ∞] A value with a name, attributed to the containing control, part, or group.

  • annotation elements [0 to ∞] A name/value pair with optional explanatory remarks.

  • link elements (markup-line) [0 to ∞] A reference to a local or remote resource

  • date-authorized element (date) [0 or 1] The date this system received its authorization.

  • security-sensitivity-level element (string) [1] The overall information system sensitivity categorization, such as defined by FIPS-199.

    The value must be one of the following:

    • low: A 'low' sensitivity level, such as defined by FIPS-199.
    • moderate: A 'moderate' sensitivity level, such as defined by FIPS-199.
    • high: A 'high' sensitivity level, such as defined by FIPS-199.

  • system-information element [1] Contains details about all information types that are stored, processed, or transmitted by the system, such as privacy information, and those defined in NIST SP 800-60.

  • security-impact-level element [1] The overall level of expected impact resulting from unauthorized disclosure, modification, or loss of access to information.

  • status element [1] Describes the operational status of the system.

  • authorization-boundary element [1] A description of this system's authorization boundary, optionally supplemented by diagrams that illustrate the authorization boundary.

  • network-architecture element [0 or 1] A description of the system's network architecture, optionally supplemented by diagrams that illustrate the network architecture.

  • data-flow element [0 or 1] A description of the logical flow of information within the system and across its boundaries, optionally supplemented by diagrams that illustrate these flows.

  • responsible-party elements [0 to ∞] A reference to a set of organizations or persons that have responsibility for performing a referenced role relative to the parent context.

  • remarks element (markup-multiline) [0 or 1] Additional commentary on the parent item.

system-security-plan

Name System Security Plan (SSP)

Description A system security plan, such as those described in NIST SP 800-18

system-security-plan is the root (containing) element of this schema.

The system-security-plan element has the following contents (in order):

  • uuid attribute (uuid) [required]

  • metadata element [1] Provides information about the publication and availability of the containing document.

  • import-profile element [1] Used to import the OSCAL profile representing the system's control baseline.

  • system-characteristics element [1] Contains the characteristics of the system, such as its name, purpose, and security impact level.

  • system-implementation element [1] Provides information as to how the system is implemented.

  • control-implementation element [1] Describes how the system satisfies a set of controls.

  • back-matter element [0 or 1] A collection of citations and resource references.

user

Name System User Class

Description A type of user that interacts with the system based on an associated role.

This element appears inside: system-implementation.

The user element has the following contents (in order):

  • uuid attribute (uuid) [required] The unique identifier for the user class.

  • title element (markup-line) [0 or 1] A title for display and navigation

  • short-name element (string) [0 or 1] A common name, short name or acronym

  • description element (markup-multiline) [0 or 1] A description supporting the parent item.

  • prop elements (string) [0 to ∞] A value with a name, attributed to the containing control, part, or group.

  • annotation elements [0 to ∞] A name/value pair with optional explanatory remarks.

  • link elements (markup-line) [0 to ∞] A reference to a local or remote resource

  • role-id elements (NCName) [1 to ∞] A reference to the roles served by the user.

    The value may be locally defined, or one of the following:

    • asset-owner: Accountable for ensuring the asset is managed in accordance with organizational policies and procedures.
    • asset-administrator: Responsible for administering a set of assets.
    • soc: Members of the security operations center (SOC).
    • noc: Members of the network operations center (NOC).
    • incident-response: Members of the incident response team.
    • help-desk: Members of the help desk.
    • configuration-management-lead: Responsible for the configuration management processes governing changes to the asset.

  • authorized-privilege elements [0 to ∞] Identifies a specific system privilege held by the user, along with an associated description and/or rationale for the privilege.

  • remarks element (markup-multiline) [0 or 1] Additional commentary on the parent item.

Remarks

Permissible values to be determined closer to the application, such as by a receiving authority.
