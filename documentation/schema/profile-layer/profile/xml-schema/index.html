OSCAL Profile Model XML Schema Reference

Schema download oscal_profile_schema.xsd

Schema version 1.0.0-milestone3

Short name oscal-profile

XML namespace http://csrc.nist.gov/ns/oscal/1.0

A profile designates a selection and configuration of controls from one or more catalogs, along with a series of operations over them. The topmost element in the OSCAL profile XML schema is profile.

add

Name Addition

Description Specifies contents to be added into controls, in resolution

This element appears inside: alter.

The add element has the following contents (in order):

  • position attribute (string) [optional] Where to add the new content with respect to the targeted element (beside it or inside it)

    The value must be one of the following:

    • before: Preceding the id-ref target
    • after: Following the id-ref target
    • starting: Inside the control or id-ref target, at the start
    • ending: Inside the control or id-ref target, at the end

  • id-ref attribute (NCName) [optional] Target location of the addition.

  • title element (markup-line) [0 or 1] A title for display and navigation

  • param elements [0 to ∞] Parameters provide a mechanism for the dynamic assignment of value(s) in a control.

  • prop elements (string) [0 to ∞] A value with a name, attributed to the containing control, part, or group.

  • annotation elements [0 to ∞] A name/value pair with optional explanatory remarks.

  • link elements (markup-line) [0 to ∞] A reference to a local or remote resource

  • part elements [0 to ∞] A partition or component of a control or part

Remarks

When no id-ref is given, the addition is inserted into the control targeted by the alteration at the start or end as indicated by position. Only position values of "starting" or "ending" are permitted when there is no id-ref.

id-ref, when given, should indicate, by its ID, an element inside the control to serve as the anchor point for the addition. In this case, position value may be any of the permitted values.

addr-line

Name Address line

Description A single line of an address.

An element (string)

This element appears inside: address.

address

Name Address

Description A postal address.

This element appears inside: location and party.

The address element has the following contents (in order):

  • type attribute (string) [optional] Indicates the type of address.

    The value may be locally defined, or one of the following:

    • home: A home address.
    • work: A work address.

  • addr-line elements (string) [0 to ∞] A single line of an address.

  • city element (string) [0 or 1] City, town or geographical region for mailing address

  • state element (string) [0 or 1] State, province or analogous geographical region for mailing address

  • postal-code element (string) [0 or 1] Postal or ZIP code for mailing address

  • country element (string) [0 or 1] Country for mailing address

algorithm

Name Hash algorithm

Description Method by which a hash is derived

An attribute (string)

The value may be locally defined, or one of the following:

  • SHA-224
  • SHA-256
  • SHA-384
  • SHA-512
  • SHA3-224
  • SHA3-256
  • SHA3-384
  • SHA3-512

This attribute appears on: hash.

Remarks

Valid values MUST be a value defined in the W3C XML Security Algorithm Cross-Reference Digest Methods (W3C, April 2013).

all

Name Include all

Description Include all controls from the imported resource (catalog)

An element (empty)

This element appears inside: include.

The all element may have the attribute:

  • with-child-controls attribute (NCName) [optional]

    The value must be one of the following:

    • yes: Include child controls with an included control.
    • no: When importing a control, only include child controls that are also explicitly called.

Remarks

This element provides an alternative to calling controls individually from a catalog. But this is also the default behavior when no include element is given in an import; so ordinarily one might not see this element unless it is for purposes of including its with-child-controls flag.

alter

Name Alteration

Description An Alter element specifies changes to be made to an included control when a profile is resolved.

This element appears inside: modify.

The alter element has the following contents (in order):

  • control-id attribute (NCName) [optional]

  • remove elements (empty) [0 to ∞] Specifies elements to be removed from a control, in resolution

  • add elements [0 to ∞] Specifies contents to be added into controls, in resolution

Remarks

Use @control-id to indicate the scope of alteration.

It is an error for two alter elements to apply to the same control. In practice, multiple alterations can be applied (together), but it creates confusion.

At present, no provision is made for altering many controls at once (for example, to systematically remove properties or add global properties); extending this element to match multiple control IDs could provide for this.

annotation

Name Annotation

Description A name/value pair with optional explanatory remarks.

This element appears inside: location, party, role, responsible-party, and add.

The annotation element has the following contents:

  • name attribute (NCName) [required]

  • uuid attribute (uuid) [required]

  • ns attribute (string) [optional]

  • value attribute (string) [optional] Indicates the value of the characteristic.

  • remarks element (markup-multiline) [0 or 1] Additional commentary on the parent item.

Remarks

Annotations permit the deployment and management of arbitrary controlled values for any purpose useful to an application or implementation scenario.

The value (flag) of an annotation is a simple scalar value or list of values. The body of the annotation offers remarks in explanation of or reference to this value.

as-is

Name As is

Description An As-is element indicates that the controls should be structured in resolution as they are structured in their source catalogs. It does not contain any elements or attributes.

An element (boolean)

This element appears inside: merge.

back-matter

Name Back matter

Description A collection of citations and resource references.

This element appears inside: profile.

The back-matter element has the following contents:

  • resource elements [0 to ∞] A resource associated with the present document, which may be a pointer to other data or a citation.

Remarks

Provides a collection of identified citation and resource objects that can be referenced by a link with a rel value of "reference" and an href value that is a fragment "#" followed by a reference to a citation identifier. A citation can reference a resource by specifying a local target value using the same fragment-identifier approach.

The following is a contrived example to show the use of link, citation, and resource.

<profile>
<metadata>
<link rel="citation" href="#resource1">My citation</link></metadata>
<back-matter>
<resource id="resource1">
<rlink href="https://example.org/some-resource"></rlink></resource></back-matter></profile>

call

Name Call

Description Call a control by its ID

An element (empty)

This element appears inside: custom, group, include, and exclude.

The call element may have attributes:

  • control-id attribute (NCName) [required]

  • with-child-controls attribute (NCName) [optional]

    The value must be one of the following:

    • yes: Include child controls with an included control.
    • no: When importing a control, only include child controls that are also explicitly called.

Remarks

If with-child-controls is yes on the call to a control, no sibling callelements need to be used to call any controls appearing within it. Since generally, this is how control enhancements are represented (as controls within controls), this provides a way to include controls with all their dependent controls (enhancements) without having to call them individually.

citation

Name Citation

Description A citation consisting of end note text and optional structured bibliographic data.

This element appears inside: resource.

The citation element has the following contents (in order):

  • text element (markup-line) [1] A line of textual content whose semantic is determined by the context of use.

  • prop elements (string) [0 to ∞] A value with a name, attributed to the containing control, part, or group.

  • biblio element [0 or 1] A container in which a set of bibliographic information can included. The model of this information is undefined by OSCAL.

Remarks

A text is used to define the endnote text, without any explicit bibliographic structure. If structured bibliographic data is needed, then the biblio> can be used for this purpose.

A biblio can be used to capture a structured bibliograpical citation in an appropriate format.

class

Name Class

Description Indicating the type or classification of the containing object

An attribute (NCName)

This attribute appears on: prop, param, part, group, and set-parameter.

Remarks

A class can be used in validation rules to express extra constraints over items of a specific class value. Some objects where this flag appears also permit a name and a ns to be assigned, which can provide for more rigor in validation.

combine

Name Combination rule

Description A Combine element defines whether and how to combine multiple (competing) versions of the same control

An element (empty)

This element appears inside: merge.

The combine element may have the attribute:

  • method attribute (string) [optional]

    The value must be one of the following:

    • use-first: Use the first definition - the first control with a given ID is used; subsequent ones are discarded
    • merge: Merge - controls with the same ID are combined
    • keep: Keep - controls with the same ID are kept, retaining the clash

Remarks

Whenever combining controls from multiple (import) pathways, an issue arises of what to do with clashing invocations (multiple competing versions of a control).

This setting permits a profile designer to apply a rule for the resolution of such cases. In a well-designed profile, such collisions would ordinarily be avoided, but this setting can be useful for defining what to do when it occurs.

custom

Name Custom grouping

Description A Custom element frames a structure for embedding represented controls in resolution.

This element appears inside: merge.

The custom element has the following contents:

  • A choice between:

    • group elements [0 to ∞] As in catalogs, a group of (selected) controls or of groups of controls

    • call elements (empty) [0 to ∞] Call a control by its ID

    • match elements (empty) [0 to ∞] Select controls by (regular expression) match on ID

Remarks

The custom element represents a custom arrangement or organization of controls in the resolution of a catalog.

While the as-is element provides for a restitution of a control set's organization (in one or more source catalogs), this element permits the definition of an entirely different structure.

doc-id

Name Document Identifier

Description A document identifier qualified by an identifier type.

An element (string)

This element appears inside: metadata and resource.

The doc-id element must have the attribute:

  • type attribute (string) [required] Qualifies the kind of document identifier.

    The value may be locally defined, or one of the following:

    • doi: A Digital Object Identifier (DOI); use is preferred, since this allows for retrieval of a full bibliographic record.

Remarks

A document identifier or name, qualified by a type that indicates the naming system of the identifier. A document identifier may provide a hint to the persistent location of the same or another version of the document.

Specific usage rules, nomenclatures, and recognized file types (e.g., DOIs, CMS locators, etc.) may be defined at application level. The value of this field may also be a hash or checksum.

Name Exclude controls

Description Which controls to exclude from the resource (source catalog) being imported

This element appears inside: import.

The exclude element has the following contents:

    • call elements (empty) [0 to ∞] Call a control by its ID

    • match elements (empty) [0 to ∞] Select controls by (regular expression) match on ID

Remarks

Within exclude, all is not an option since it makes no sense. However, it also makes no sense to use exclude/call except with include/all; you would not want to include and exclude something by ID simultaneously. If this happens, an error condition will be reported.

hash

Name Hash

Description A representation of a cryptographic digest generated over a resource using a hash algorithm.

An element (string)

This element appears inside: rlink.

The hash element must have the attribute:

  • algorithm attribute (string) [required]

    The value may be locally defined, or one of the following:

    • SHA-224
    • SHA-256
    • SHA-384
    • SHA-512
    • SHA3-224
    • SHA3-256
    • SHA3-384
    • SHA3-512

Remarks

A hash value can be used to authenticate that a referenced resource is the same resources as was pointed to by the author of the reference.

When appearing as part of a resource (data object), the hash should apply to the referenced object.

An attribute (uri-reference)

id

Name Identifier

Description Unique identifier of the containing object

An attribute (NCName)

This attribute appears on: role, param, usage, part, and group.

Remarks

No mechanism is proposed to ensure that id values do not collide across different catalogs. Use profiling without merge to detect such clashes.

import

Name Import resource

Description An Import element designates a catalog, profile, or other resource to be included (referenced and potentially modified) by this profile.

This element appears inside: profile.

The import element has the following contents (in order):

  • href attribute (uri-reference) [required]

  • include element [0 or 1] Specifies which controls to include from the resource (source catalog) being imported

  • exclude element [0 or 1] Which controls to exclude from the resource (source catalog) being imported

Remarks

An import indicates a source whose controls are to be included (referenced and modified) in a profile. This source will either be a catalog whose controls are given (by value), or a profile with its own control imports (with possible settings.

The contents of the import element indicate which controls from the source will be included. Controls may be either selected (using an include element) or de-selected (using an exclude element) from the source catalog or profile.

When no include is given (whether an exclude is given or not), an include/all is assumed (that is, all controls will be included by default).

<import href="catalog.xml">
<include>
<call control-id="ac-1"></call></include></import>

include

Name Include controls

Description Specifies which controls to include from the resource (source catalog) being imported

This element appears inside: import.

The include element has the following contents:

  • A choice between:

    • all element (empty) [0 or 1] Include all controls from the imported resource (catalog)

    • call elements (empty) [0 to ∞] Call a control by its ID

    • match elements (empty) [0 to ∞] Select controls by (regular expression) match on ID

Remarks

To be schema-valid, this element must contain either (but not both) a single all element, or a sequence of call elements.

If this element is not given, it is assumed to be present with contents all (qv); i.e., all controls are included by default, unless the include instruction calls controls specifically.

label

Name Parameter label

Description A placeholder for a missing value, in display.

An element (markup-line)

This element appears inside: param and set-parameter.

last-modified

Name Last modified timestamp

Description Date and time of last modification.

An element (dateTime-with-timezone)

This element appears inside: metadata and revision.

Remarks

This value represents the point in time when the OSCAL document was last updated. The date value must be an ISO 8601 formatted date with full time and time zone indicated. Ordinarily, a date value will be machine generated at time of creation or modification.

In some cases, an OSCAL document may be derived from some source material in a different format. In such a case, the last-modified value should indicate the modification time of the OSCAL document, not the source material.

link

Name Link

Description A reference to a local or remote resource

An element (markup-line)

This element appears inside: metadata, revision, location, party, role, responsible-party, param, part, set-parameter, and add.

The link element may have attributes:

  • href attribute (uri-reference) [required]

  • rel attribute (NCName) [optional]

    The value may be locally defined, or one of the following:

    • homepage: A link to a homepage related to the containing object.
    • interview-notes: Interview Notes
    • tool-output: Tool Output
    • photograph: Photograph
    • questionnaire: Questionnaire
    • screen-shot: Screen Shot

  • media-type attribute (string) [optional]

Remarks

This is a rough analog to an HTML anchor (a), except this is a line-oriented (block) element.

The href must point to a local or remote resource. If a local resource is used, this will be indicated by a fragment "#" followed by an identifier which references an identified citation or resource in the document's back-matter. If a remote resource is used, the href value will be an absolute or relative URI pointing to the location of the referenced resource. A relative URI can be resolved relative to the location of the document containing the link.

To provide a cryptographic hash for a remote target resource, a local reference to a resource is needed. The resource allows one or more hash values to be provided using the rlink hash object.

The media-type provides a hint about the content model of the referenced resource. A valid entry from the IANA Media Types registry

The following is a contrived example to show the use of link, citation, and resource.

<oscal>
            
<link rel="reference" href="#resource1">My Hashed Resource</link>
        ...snip...
        
<back-matter>
               
<resource id="resource1">
                  
<rlink href="https://example.org/some-resource">
                     
<hash algorithm="sha512">C2E9C1..snip..F88D2E</hash>
                  </rlink>
               </resource>
            </back-matter>
         </oscal>

merge

Name Merge controls

Description A Merge element merges controls in resolution.

This element appears inside: profile.

The merge element has the following contents (in order):

  • combine element (empty) [0 or 1] A Combine element defines whether and how to combine multiple (competing) versions of the same control

  • A choice between:

    • as-is element (boolean) [0 or 1] An As-is element indicates that the controls should be structured in resolution as they are structured in their source catalogs. It does not contain any elements or attributes.

    • custom element [0 or 1] A Custom element frames a structure for embedding represented controls in resolution.

Remarks

The contents of the merge element may be used to reorder or restructure controls by indicating an order and/or structure in resolution.

Implicitly, a merge element is also a filter: controls that are included in a profile, but not included (implicitly or explicitly) in the scope of a merge element, will not be merged into (will be dropped) in the resulting resolution.

  • responsible-party elements [0 to ∞] A reference to a set of organizations or persons that have responsibility for performing a referenced role relative to the parent context.

  • remarks element (markup-multiline) [0 or 1] Additional commentary on the parent item.

name

Name Name

Description Identifying the purpose and intended use of the property, part or other object.

An attribute (NCName)

This attribute appears on: prop, annotation, and part.

Remarks

This flag is intended to be used as a control point for validation of properties and parts with respect to constraints related to local semantics. A name is not necessarily expected to be unique, even within the scope of the parent (containing) structure. In some local applications, it is possible for a name to be unique and uniqueness of a name can be validated in such cases.

The definition of a name is subject to its namespace, offered by an ns flag (q.v.). When ns is not given, the namespace is assumed to be oscal and the name should be one of the recognized values for names in OSCAL.

Name Namespace

Description A namespace qualifying the name.

An attribute (string)

This attribute appears on: prop, annotation, and part.

Remarks

Identifies an organization or compliance regime defining the information needs (requirements) addressed by an object. When this attribute is not given, its value should be assumed to be oscal and the name should be a name defined by this specification.

param

Name Parameter

Description Parameters provide a mechanism for the dynamic assignment of value(s) in a control.

This element appears inside: group and add.

The param element has the following contents (in order):

  • id attribute (NCName) [required]

  • class attribute (NCName) [optional]

  • depends-on attribute (NCName) [optional]

  • label element (markup-line) [0 or 1] A short name for the parameter.

    The label value should be suitable for inline display in a rendered catalog.

  • usage elements (markup-line) [0 to ∞] Indicates and explains the purpose and use of a parameter

  • constraint elements (string) [0 to ∞] A formal or informal expression of a constraint or test

  • guideline elements [0 to ∞] A prose statement that provides a recommendation for the use of a parameter.

  • A choice between:

    • value element (markup-line) [0 or 1] A recommended parameter value or set of values.

      A value provided in a catalog can be redefined at any higher layer of OSCAL (e.g., Profile).

    • select element [0 or 1] A set of parameter value choices, that may be picked from to set the parameter value.

      .

  • link elements (markup-line) [0 to ∞] A reference to a local or remote resource

  • Any element (in a foreign namespace)

Remarks

In a catalog, a parameter is typically used as a placeholder for the future assignment of a parameter value, although the OSCAL model allows for the direct assignment of a value if desired by the control author. The value may be optionally used to specify one or more values. If no value is provided, then it is expected that the value will be provided at the Profile or Implementation layer.

A parameter can include a variety of metadata options that support the future solicitation of one or more values. A label provides a textual placeholder that can be used in a tool to solicit parameter value input, or to display in catalog documentation. The desc provides a short description of what the parameter is used for, which can be used in tooling to help a user understand how to use the parameter. A constraint can be used to provide criteria for the allowed values. A guideline provides a recommendation for the use of a parameter.

part

Name Part

Description A partition or component of a control or part

This element appears inside: part, group, and add.

The part element has the following contents (in order):

  • id attribute (NCName) [optional]

  • name attribute (NCName) [required]

  • ns attribute (string) [optional]

  • class attribute (NCName) [optional]

  • title element (markup-line) [0 or 1] A title for display and navigation

  • prop elements (string) [0 to ∞] A value with a name, attributed to the containing control, part, or group.

  • prose element (markup-multiline) [0 or 1] Prose permits multiple paragraphs, lists, tables etc.

  • part elements [0 to ∞] A partition or component of a control or part

  • link elements (markup-line) [0 to ∞] A reference to a local or remote resource

  • Any element (in a foreign namespace)

Remarks

A part provides for logical partitioning of prose, and can be thought of as a grouping structure (e.g., section). A part can have child parts allowing for arbitrary nesting of prose content (e.g., statement hierarchy). A part can contain prop objects that allow for enriching prose text with structured name/value information.

A part can be assigned an optional id, which allows for internal and external references to the textual concept contained within a part. A id provides a means for an OSCAL profile, or a higher layer OSCAL model to reference a specific part within a catalog. For example, an id can be used to reference or to make modifications to a control statement in a profile.

Use of part and prop provides for a wide degree of extensibility within the OSCAL catalog model. The optional ns provides a means to qualify a part's name, allowing for organization-specific vocabularies to be defined with clear semantics. Any organization that extends OSCAL in this way should consistently assign a ns value that represents the organization, making a given namespace qualified name unique to that organization. This allows the combination of ns and name to always be unique and unambiguous, even when mixed with extensions from other organizations. Each organization is responsible for governance of their own extensions, and is strongly encouraged to publish their extensions as standards to their user community. If no ns is provided, the name is expected to be in the "OSCAL" namespace.

To ensure a ns is unique to an organization and naming conflicts are avoided, a URI containing a DNS or other globally defined organization name should be used. For example, if FedRAMP and DoD both extend OSCAL, FedRAMP will use the ns "https://fedramp.gov", while DoD will use the ns "https://defense.gov" for any organization specific name.

Tools that process OSCAL content are not required to interpret unrecognized OSCAL extensions; however, OSCAL-compliant tools should not modify or remove unrecognized extensions, unless there is a compelling reason to do so, such as data sensitivity.

<part name="statement" id="statement-A">
<part ns="https://fedramp.gov" name="status" id="statement-A-FedRAMP">Something FedRAMP Cares About</part>
<part ns="https://defense.gov" name="status" id="statement-A-DoD">Something DoD Cares About</part></part>

profile

Name Profile

Description Each OSCAL profile is defined by a Profile element

profile is the root (containing) element of this schema.

The profile element has the following contents (in order):

  • uuid attribute (uuid) [required]

  • metadata element [1] Provides information about the publication and availability of the containing document.

  • import elements [1 to ∞] An Import element designates a catalog, profile, or other resource to be included (referenced and potentially modified) by this profile.

  • merge element [0 or 1] A Merge element merges controls in resolution.

  • modify element [0 or 1] Set parameters or amend controls in resolution

  • back-matter element [0 or 1] A collection of citations and resource references.

Remarks

An OSCAL document that describes a selection with possible modification of multiple controls from multiple catalogs. It provides mechanisms by which controls may be selected (import), merged or (re)structured (merge), and emended (modify). OSCAL profiles may select subsets of control sets, set parameter values for them in application, and even qualify the representation of controls as given in and by a catalog. They may also serve as sources for further modification in and by other profiles, that import them.

prop

Name Property

Description A value with a name, attributed to the containing control, part, or group.

An element (string)

This element appears inside: metadata, revision, location, party, resource, citation, role, responsible-party, part, group, and add.

The prop element may have attributes:

  • name attribute (NCName) [required]

    The value may be locally defined, or one of the following:

    • marking: A label or descriptor that is tied to a sensitivity or classification marking system. An optional class can be used to define the specific marking system used for the associated value.

  • uuid attribute (uuid) [required]

  • ns attribute (string) [optional]

  • class attribute (NCName) [optional]

Remarks

Properties permit the deployment and management of arbitrary controlled values, with and among control objects (controls, parts, and extensions), for any purpose useful to an application or implementation of those controls. Typically, properties will be used to sort, select, order, and arrange controls or relate them to one another or to class hierarchies, taxonomies, or external authorities.

The value of a property is a simple scalar value or list of values.

The lexical composition of properties may be constrained by external processes to ensure consistency.

Because properties are often used as selectors or identifiers for OSCAL operations, their values can be expected frequently to be normalized (e.g., with respect to whitespace) in use; however, this is application defined.

published

Name Publication Timestamp

Description The date and time this document was published.

An element (dateTime-with-timezone)

This element appears inside: metadata and revision.

Remarks

This value represents the point in time when the OSCAL document was published. The date value must be an ISO 8601 formatted date with full time and time zone indicated. Ordinarily, a date value will be machine generated at time of creation or modification.

In some cases, an OSCAL document may be derived from some source material in a different format. In such a case, the last-modified-date value should indicate the modification time of the OSCAL document, not the source material. (The nominal publication date of the original source material can be captured as a named property or custom metadata construct.)

A publisher of OSCAL content can use this data point along with its sibling last-modified to establish commonality across OSCAL instances that are intended to be considered as successive current versions of a given publication.

resource

Name Resource

Description A resource associated with the present document, which may be a pointer to other data or a citation.

This element appears inside: back-matter.

The resource element has the following contents (in order):

  • uuid attribute (uuid) [required]

  • title element (markup-line) [0 or 1] A title for display and navigation

  • desc element (string) [0 or 1] A short textual description

  • prop elements (string) [0 to ∞] A value with a name, attributed to the containing control, part, or group.

  • doc-id elements (string) [0 to ∞] A document identifier qualified by an identifier type.

  • citation element [0 or 1] A citation consisting of end note text and optional structured bibliographic data.

  • rlink elements [0 to ∞] A pointer to an external copy of a document with optional hash for verification

  • base64 elements (base64Binary) [0 to ∞]

  • remarks element (markup-multiline) [0 or 1] Additional commentary on the parent item.

  • Any element (in a foreign namespace)

Remarks

Use of resource allows for information that pertains to, authenticates, or supplements the document it is included in.

A resource can be used in two ways. 1) it may point to an specific retrievable network resource using a rlink, and 2) be included as an attachment using a base64. A resource may contain multiple rlink and base64 entries that represent alternative download locations (rlink) and attachments (base64) for the same resource. Both rlink and base64 allow for a media-type to be specified, which is used to distiguish between different representations of the same resource (e.g., Microsoft Word, PDF). When multiple rlink and base64 entries are included for a given resource, all entries must contain equivalent information. This is extremely important when the entries represent OSCAL content that is represented in alternate formats (i.e., XML, JSON, YAML). A processor must be able to choose a preferred format to process (based on media-type) with the assumption that the same data can be processed from any of the available formats included by entries.

When a resource is used as a citation, then the title and citation properties must be included.
